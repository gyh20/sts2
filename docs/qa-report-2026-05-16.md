# STS2 Bingo QA report - 2026-05-16

## Scope

Checked the current `/sts2` implementation with a separate reviewer pass plus local automated tests against a temporary copy of the service.

## Automated checks

- Syntax: `node --check server.js`, `node --check public/app.js`
- Smoke: `npm test`
- API/feature test run against an isolated `/tmp` copy on port 8791

Result: **124/124 checks passed** after fixes.

Covered areas:

- register / login / logout / current user
- room creation and lookup
- join room, max 4 players
- host-only settings permissions
- team switching in lobby
- normal mode seed matching and win rejection
- wrong seed rejection
- 5 minute submit cooldown
- same-team submission privacy before finish
- public submission visibility after finish
- all 56 catalog tasks with synthetic pass and fail samples
- malformed URL and malformed cookie hardening
- `/sts2` public page and `/sts2/api/tasks` production smoke check

## Fixes made from review

- Malformed URL / malformed cookie no longer crash the service.
- Room IDs are now generated from a safe uppercase alphanumeric alphabet.
- Submissions are accepted only while the room status is `playing`.
- Start / restart clears board state, aggregate task state, and prior room submissions.
- Players cannot join or switch teams after the game starts.
- Session cookie path is scoped to `/sts2`.
- Static file path containment now uses resolved paths.
- Character-specific seeds fail if the submitted save lacks a character field.
- Basic write operations are serialized in-process to reduce lockout / DB write races.
- Frontend route loading and copy-link fallback are more robust.

## Remaining known limitations

- Save files are plain JSON and can be hand-edited. This is acceptable for casual play, but not secure for a serious competition unless STS2 provides a trusted signature or another external verification source.
- The file DB is now serialized within one Node process, but SQLite would still be safer for long-term multi-user use.
- `no_rare_card_pick` is still a weak evaluator because the current save samples do not reliably expose card rarity for every card reward. It currently behaves as a known limitation rather than a fully trustworthy task.
- Some task semantics may need tuning once more real STS2 save examples are available, especially multi-player save files and any future task that depends on card/relic rarity.
